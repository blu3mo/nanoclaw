# 朝のセッション詳細手順

このファイルは朝のスケジュールタスク実行時に Blueclaw が参照する進行マニュアルである。

---

## ステップ1: 体調・エネルギーの確認

- 「今日の調子はどう？エネルギーは1-10で？」と聞く
- 体調が悪い、元気がない場合はそれを正直に申告させる
- **無理にできないことを毎回宣言させても意味がない。** 今日のコンディションに合った現実的な計画を立てさせる
- ただし「体調が悪い」が特定タスクの前に毎回出る場合はパターンとして USER.md に記録する（→ 回避シグナル）

## ステップ2: 今日やること3つの宣言

- **最大3つ**に絞らせる（全部書いてもどうせやれないので）
- 各タスクについて以下を言語化させる:
  - **今日の達成条件** — 何ができたら「やった」と言えるか
  - **すぐ手を動かせるレベルのネクストアクション**
  - **ネクストアクションの「最初の10秒でやること」** — 身体的動作レベル（「ターミナルを開く」「ブラウザで〜を開く」等）
  - **いつやるのか** — 時間ブロック（実際にカレンダー上で時間をブロックさせる）
  - **見積もり時間**

## ステップ3: 疑いの目で検証する

ユーザーの言うことを鵜呑みにしない。自分に甘くなる傾向がある。

- 「それ本当に今日やれる？」 — 過去の失敗パターン（daily/ の記録）を踏まえて疑ってかかる
- 「この見積もりの根拠は？」 — 曖昧なら具体化させる
- 「前回同じことを宣言してやれなかったけど、今回は何が違う？」
- エネルギーレベルが低い場合は特に、宣言が過大でないかチェック
- 必要に応じてスコープを絞らせる。**少なく宣言して確実にやる方がいい**

- 「この3つの中で一番大事なのはどれ？」
- 高エネルギー時間帯に最も認知負荷の高いタスクを配置させる
- タスク間の切り替えコスト（15-20分）を計算に入れさせる

## ステップ5: チェックインのスケジュール登録

朝のセッション完了後、宣言された時間ブロックに基づいてチェックインを登録する。

例: ユーザーが「10:00-11:00にタスクA、14:00-15:30にタスクB」と宣言した場合:

```
schedule_task:
  prompt: "タスクAの開始チェックイン。「タスクAの時間。始めた？」と聞く。返信がなければ10分後にフォローアップの once タスクを登録する。"
  schedule_type: once
  schedule_value: "2026-04-05T10:00:00"  ← 当日の日付を使う
  context_mode: group

schedule_task:
  prompt: "タスクAの終了チェックイン。「タスクAどうだった？何ができた？」と聞く。完了報告を受けたら daily/ に記録。詰まっていたらブロッカー分析。"
  schedule_type: once
  schedule_value: "2026-04-05T11:00:00"
  context_mode: group
```

タスクBについても同様に登録する。

**重要**: `schedule_value` にはローカル時間（Zサフィックスなし）を使うこと。

## ステップ6: 記録

朝のセッションの結果を `daily/YYYY-MM-DD.md` に記録する:

```markdown
# YYYY-MM-DD

## 朝の宣言
- エネルギー: X/10
- タスク1: [名前] — 達成条件: [条件] — 時間: [HH:MM-HH:MM]
- タスク2: ...
- タスク3: ...

## 実績
（夜のセッションで記入）

## 振り返り
（夜のセッションで記入）
```

# Blueclaw 実装方針 — NanoClawベースの設計提案

> このドキュメントは、DESIGN.md に記述された Blueclaw の挙動仕様を NanoClaw 上で実現するための技術的な方針を詳細に説明する。

---

## 1. NanoClaw のアーキテクチャ概要（まず理解すべきこと）

### 1.1 全体像

NanoClaw は「チャットアプリ経由で Claude に指示を出し、結果を受け取る」ためのシステム。以下の3層で構成される。

```
┌─────────────────────────────────────────────────────┐
│ チャットチャネル（WhatsApp, Telegram, Slack 等）       │
│   ユーザーがメッセージを送る場所                        │
└──────────────────────┬──────────────────────────────┘
                       │ メッセージ保存 / 取得
                       ▼
┌─────────────────────────────────────────────────────┐
│ ホストプロセス（Node.js 単一プロセス）                  │
│   src/index.ts が全体を統括                           │
│   - チャネル接続・メッセージ受信                        │
│   - SQLite にメッセージ保存                            │
│   - ポーリングループで新メッセージ検知                    │
│   - コンテナ起動・IPC管理                              │
│   - スケジュールタスク実行                              │
└──────────────────────┬──────────────────────────────┘
                       │ docker run / stdin・stdout
                       ▼
┌─────────────────────────────────────────────────────┐
│ コンテナ（Linux VM 内の隔離環境）                      │
│   Claude Agent SDK (claude-code) が実行される          │
│   - container/agent-runner/src/index.ts              │
│   - MCP サーバー（send_message, schedule_task 等）     │
│   - グループフォルダのみマウント（ファイルシステム隔離）    │
│   - groups/{name}/CLAUDE.md がエージェントの指示書       │
└─────────────────────────────────────────────────────┘
```

### 1.2 メッセージの流れ（受信→処理→返信）

1. **受信**: チャネル（WhatsApp等）がメッセージを受信 → `onMessage` コールバック → `storeMessage()` で SQLite に保存
2. **検知**: `startMessageLoop()` が 2秒間隔でポーリング → 登録済みグループの新メッセージを検出
3. **トリガー判定**: メインチャネルなら全メッセージ処理。それ以外はトリガーワード（例: `@Andy`）の有無を確認
4. **フォーマット**: `formatMessages()` で XML 形式に整形（送信者名、タイムスタンプ付き）
5. **コンテナ起動**: `runContainerAgent()` が Docker コンテナを起動。stdin に JSON（プロンプト、セッションID等）を渡す
6. **エージェント実行**: コンテナ内で `agent-runner/src/index.ts` が Claude Agent SDK の `query()` を呼ぶ。Claude が応答を生成
7. **結果返送**: stdout に `OUTPUT_START_MARKER` / `OUTPUT_END_MARKER` で囲んだ JSON を出力。ホストがパースして `channel.sendMessage()` で返信

### 1.3 グループシステム

- 各グループは `groups/{folder}/` に独自のフォルダを持つ
- `groups/{folder}/CLAUDE.md` がそのグループ専用のエージェント指示書
- `groups/global/CLAUDE.md` が全グループ共通の指示（読み取り専用でマウント）
- `groups/main/CLAUDE.md` がメインチャネル（管理者権限あり）の指示書
- コンテナには自分のグループフォルダしか見えない（ファイルシステム隔離）

### 1.4 スケジュールタスク

- SQLite の `scheduled_tasks` テーブルにタスクを保存
- `startSchedulerLoop()` が 60秒間隔でポーリング → 実行時刻を過ぎたタスクを検出
- タスク実行時は通常のメッセージ処理と同じくコンテナを起動して Claude に処理させる
- cron 式（`0 9 * * *` = 毎朝9時）、interval（ミリ秒間隔）、once（一回限り）に対応

### 1.5 IPC（プロセス間通信）

- コンテナ内のエージェントがホストと通信する仕組み
- MCP サーバー（`ipc-mcp-stdio.ts`）が `send_message`, `schedule_task`, `register_group` 等のツールを提供
- エージェントがこれらのツールを呼ぶと、`/workspace/ipc/` 以下に JSON ファイルが書き込まれる
- ホストの `startIpcWatcher()` が 1秒間隔でポーリングしてこれらを処理

### 1.6 セッション管理

- 各グループはセッションID を持ち、会話の文脈を維持
- Claude Agent SDK の `resume` オプションで前回のセッションを再開
- セッションが長くなると自動コンパクション（要約して圧縮）が発動
- コンパクション前にトランスクリプトを `conversations/` に保存

## 2. Blueclaw を NanoClaw 上で実現する方針

### 2.1 基本戦略

Blueclaw の機能は大きく分けて以下の3カテゴリに分類できる。それぞれの実現方法が異なる。

| カテゴリ | 実現方法 |
|---------|---------|
| **パーソナリティ・口調・行動原則** | `CLAUDE.md` への記述（コード変更不要） |
| **定期チェックイン・日次/週次サイクル** | スケジュールタスク（コード変更不要） |
| **ストック情報管理（kanban.md 等）** | グループフォルダ内のファイル管理（コード変更不要） |

**重要な結論: NanoClaw のコードをほぼ変更せずに Blueclaw を実現できる。** 変更が必要なのは主に設定と CLAUDE.md の内容。

### 2.2 アーキテクチャ全体図

```
┌─────────────────────────────────┐
│  WhatsApp（またはTelegram等）     │  ← ユーザーがメッセージを送る場所
│  自分宛セルフチャット = メイン      │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│  NanoClaw ホスト（macOS）         │
│  - メッセージ受信・保存            │
│  - ポーリング → コンテナ起動       │
│  - スケジュールタスク管理          │
└──────────────┬──────────────────┘
               │
               ▼
┌─────────────────────────────────┐
│  コンテナ（Blueclaw エージェント） │
│                                  │
│  CLAUDE.md = Blueclaw の全人格    │
│  ├── パーソナリティ・口調          │
│  ├── ADHD対応行動原則             │
│  ├── 朝/夜セッションの進行手順     │
│  └── ストック情報管理ルール        │
│                                  │
│  ファイル群:                      │
│  ├── kanban.md                   │
│  ├── USER.md                     │
│  ├── daily/YYYY-MM-DD.md         │
│  ├── weekly/YYYY-MM-DD.md        │
│  └── conversations/              │
│                                  │
│  MCP ツール:                      │
│  ├── send_message（即時送信）      │
│  ├── schedule_task（タスク管理）   │
│  └── list_tasks / cancel_task    │
└─────────────────────────────────┘
```

## 3. 具体的な実装手順

### ステップ 1: NanoClaw をセットアップする

まだセットアップしていない場合:

```bash
cd nanoclaw
claude     # Claude Code を起動
/setup     # セットアップスキルを実行
```

`/setup` が依存関係のインストール、WhatsApp 認証（QR コード）、コンテナビルド、サービス登録を全部やってくれる。

### ステップ 2: アシスタント名を「Blueclaw」に変更

`.env` ファイルで:

```
ASSISTANT_NAME=Blueclaw
```

これにより:
- トリガーワードが `@Blueclaw` になる
- レスポンスのプレフィックスが `Blueclaw:` になる
- CLAUDE.md テンプレートの名前が変わる

### ステップ 3: グループ用 CLAUDE.md を作成（最重要）

`groups/main/CLAUDE.md` を Blueclaw の全人格・行動原則を記述したものに書き換える。これが Blueclaw の「脳」になる。

以下の構成で書く:

```markdown
# Blueclaw

あなたは Blueclaw、ADHDの実行機能を外部から補完する常駐型AIマネージャーである。

## 基本姿勢
（DESIGN.md セクション3「Blueclawとユーザーの関係」の内容）

## ストック情報管理ルール
（DESIGN.md セクション4 の内容 — kanban.md, USER.md の構造と更新ルール）

## 朝のセッション進行手順
（DESIGN.md セクション5.1 の全ステップ）

## 作業中チェックインルール
（DESIGN.md セクション5.2 の内容）

## 夜のセッション進行手順
（DESIGN.md セクション5.3 の全ステップ）

## 週次セッション
（DESIGN.md セクション5.4 の内容）

## ADHD対応行動原則
（DESIGN.md セクション6 の全内容 — 回避検知、段階的押し返し等）

## モチベーション管理
（DESIGN.md セクション7 の全内容）

## 思考のリミッター解除
（DESIGN.md セクション8 の全内容）

## トーン & パーソナリティ
（DESIGN.md セクション10 の全内容 — 口調、言ってはいけないこと等）

## エネルギー管理
（DESIGN.md セクション11 の全内容）

## ファイル構成
- `kanban.md` — タスクの全体像（ゴール状態、現在の状態、ネクストアクション）
- `USER.md` — ユーザーの特性・有効な手法の蓄積知識
- `daily/YYYY-MM-DD.md` — 日次記録
- `weekly/YYYY-MM-DD.md` — 週次記録
- `conversations/` — 過去の会話ログ

## チェックイン応答パターン
（DESIGN.md セクション9 の表）
```

**注意**: CLAUDE.md は Claude のコンテキストウィンドウに毎回ロードされるため、簡潔さと網羅性のバランスが重要。DESIGN.md の内容をそのまま貼り付けると長すぎる可能性がある。要点を凝縮しつつ、判断基準（段階的押し返しの Level 1-4 等）は具体的に残す。

長くなりすぎる場合は、CLAUDE.md を主要指示に絞り、詳細マニュアルを別ファイル（例: `blueclaw-manual.md`）に分割してコンテキスト内で参照させることもできる。

### ステップ 4: スケジュールタスクで定期チェックインを実装

Blueclaw の日次サイクルは NanoClaw のスケジュールタスク機能で実現する。初回は手動でメインチャネルから指示するか、CLAUDE.md に「初回起動時にこれらのタスクをスケジュールせよ」と書いておく。

#### 朝のセッション（例: 毎朝 8:00）

```
@Blueclaw 毎朝8:00に朝のセッションを開始するタスクをスケジュールして。
プロンプト: 「朝のセッションを開始する。まずユーザーの体調・エネルギーを確認し、
今日やること3つの宣言を引き出し、見積もりを検証し、優先順位を確認する。
CLAUDE.md の朝のセッション手順に従って進行すること。」
```

NanoClaw が内部的にやること:
1. エージェントが `schedule_task` MCP ツールを呼ぶ
2. IPC 経由でホストに `schedule_task` JSON が書かれる
3. ホストが SQLite に `cron: "0 8 * * *"` でタスクを登録
4. 毎朝 8:00 にスケジューラがタスクを検出 → コンテナ起動 → Claude が朝のセッションのメッセージを送信

#### 夜のセッション（例: 毎晩 23:00）

```
@Blueclaw 毎晩23:00に夜の振り返りセッションを開始するタスクをスケジュールして。
```

#### 作業中チェックイン

朝のセッションで宣言された作業スケジュールに基づき、Blueclaw 自身がチェックインタスクを動的にスケジュールする。例えば「10:00-11:00にXをやる」と宣言されたら:

- 10:00 に `once` タスク: 「Xの時間。始めた？」
- 11:00 に `once` タスク: 「Xどうだった？何ができた？」

これは CLAUDE.md に「朝のセッション完了後、宣言されたスケジュールに基づいてチェックインタスクを schedule_task で登録すること」と指示すれば実現できる。

#### 返信がない場合の追跡

DESIGN.md には「10分後に再送、その後20-30分間隔で追跡」と書かれている。これも同様に:
- チェックインメッセージ送信後、「10分後にフォローアップ」の `once` タスクを登録
- そのタスク実行時に、ユーザーが返信済みか（kanban.md や daily/ の更新有無で判断）を確認
- 未返信なら追加のリマインダーを送信し、次のフォローアップをスケジュール

### ステップ 5: ストック情報ファイルを初期化

グループフォルダ内に初期ファイルを作成する。これは Blueclaw に指示するだけでよい:

```
@Blueclaw kanban.md, USER.md, daily/ フォルダ, weekly/ フォルダを初期化して
```

エージェントが `groups/main/` 内（コンテナ内では `/workspace/group/`）に:
- `kanban.md` — 空のテンプレート
- `USER.md` — 空のテンプレート
- `daily/` — ディレクトリ作成
- `weekly/` — ディレクトリ作成

を作る。以後、Blueclaw は各セッションでこれらのファイルを読み書きする。

### ステップ 6: Google Calendar 連携（オプション）

DESIGN.md のステップ2で「カレンダー上で時間をブロックさせる」とある。NanoClaw は MCP サーバーとして Google Calendar を統合できる（`mcp__claude_ai_Google_Calendar__*` ツールが利用可能）。

コンテナ内のエージェントから Google Calendar にアクセスするには、コンテナの MCP サーバー設定に Google Calendar を追加するか、ホスト側の MCP サーバーを IPC 経由で呼び出す仕組みが必要。

**現時点での最もシンプルなアプローチ**: カレンダーブロックはユーザーに口頭で指示する（「10:00-11:00をカレンダーにブロックして」）。将来的にカレンダー API を直接操作したい場合はコンテナの MCP 設定を拡張する。

## 4. コード変更が必要な箇所（最小限）

### 4.1 基本的にコード変更は不要

上記の方針で Blueclaw の中核機能（パーソナリティ、日次サイクル、ストック情報管理）はすべて **CLAUDE.md とスケジュールタスクだけで実現できる**。NanoClaw のソースコードを変更する必要はない。

### 4.2 あると便利な拡張（任意）

以下は「あるとより良い」拡張であり、必須ではない:

#### a) コンテキストモードのデフォルト変更

Blueclaw のチェックインタスクは前回の会話文脈が必要なので、`context_mode: "group"` で実行する必要がある。これはタスクスケジュール時に指定すればよいので、コード変更は不要。

#### b) メッセージ未読検知

「返信がない場合の追跡」で、ユーザーが返信したかどうかを検知する必要がある。現状のNanoClawでもこれは可能:
- スケジュールタスクが `context_mode: "group"` で実行されれば、前回の会話が見える
- タスク実行時に「最後のメッセージが Blueclaw のチェックインのままか」を確認できる
- エージェント自身が SQLite を直接読むこともできる（メインチャネルは store/ がマウントされている）

#### c) タスクスクリプトによる軽量チェック

頻繁なチェックインでAPIコストを抑えたい場合、`script` オプションで「ユーザーが返信済みかをSQLiteで確認 → 未返信なら wakeAgent: true」というスクリプトを使える:

```bash
# チェックイン後にユーザーが返信したか確認
LAST_BOT_TS=$(sqlite3 /workspace/project/store/messages.db \
  "SELECT MAX(timestamp) FROM messages WHERE chat_jid='$CHAT_JID' AND is_bot_message=1")
USER_REPLY=$(sqlite3 /workspace/project/store/messages.db \
  "SELECT COUNT(*) FROM messages WHERE chat_jid='$CHAT_JID' AND timestamp>'$LAST_BOT_TS' AND is_bot_message=0")
if [ "$USER_REPLY" -gt 0 ]; then
  echo '{"wakeAgent": false}'
else
  echo '{"wakeAgent": true, "data": {"reason": "no reply after checkin"}}'
fi
```

ただし、これはメインチャネルのみ（store/ がマウントされているのはメインのみ）。Blueclaw はメインチャネル（セルフチャット）での1対1利用を想定しているので問題ない。

## 5. デプロイ方法

### 5.1 ローカル（macOS）— 推奨

NanoClaw はバックグラウンドサービスとして常時起動する。

```bash
# セットアップ（初回のみ）
cd nanoclaw
claude
/setup

# サービス管理
launchctl load ~/Library/LaunchAgents/com.nanoclaw.plist      # 起動
launchctl unload ~/Library/LaunchAgents/com.nanoclaw.plist    # 停止
launchctl kickstart -k gui/$(id -u)/com.nanoclaw              # 再起動
```

`/setup` が `com.nanoclaw.plist` を自動生成し、macOS のログイン時に自動起動するよう設定する。

**macOS がスリープするとどうなるか**: launchd はスリープ中はタスクを実行しない。復帰後にスケジューラが due タスクを検出して実行する。つまりスリープ中のチェックインは送られないが、復帰後にまとめて処理される。常時稼働が必要なら下記のクラウドデプロイを検討。

### 5.2 クラウド（VPS）— 常時稼働が必要な場合

スリープせず24時間稼働させたい場合:

1. Linux VPS（例: AWS EC2, DigitalOcean, Hetzner）を用意
2. Docker をインストール
3. リポジトリをクローンして `/setup` を実行
4. systemd でサービス化:

```bash
systemctl --user start nanoclaw
systemctl --user enable nanoclaw  # 自動起動
```

**注意点**:
- WhatsApp 認証はローカルで QR コードスキャンが必要。ローカルで認証後、認証データ (`data/whatsapp-auth/`) をサーバーにコピーするか、ペアリングコード方式を使う
- Anthropic API キーと OneCLI の設定が必要
- コンテナ内コンテナ（Docker-in-Docker）ではなく、ホストの Docker デーモンを使う

### 5.3 推奨: まずローカルで動かす

Blueclaw の挙動を調整・テストするにはローカルの方が圧倒的に楽。CLAUDE.md の調整、スケジュールの調整、口調の微調整を繰り返すことになるので、まずはローカルで安定させてからクラウドに移行するのが合理的。

1. WhatsApp のセルフチャット（メインチャネル）を開く
2. `@Blueclaw 初めまして。セットアップを開始して。` と送る
3. Blueclaw が CLAUDE.md の指示に従い、以下を実行:

   - 初回の朝のセッションを開始（今日やること3つの宣言）

## 7. 設計上の考慮事項

### 7.1 コンテキストウィンドウの管理

Blueclaw はセッション間で文脈を維持する必要がある。NanoClaw はこれを以下の仕組みで対処:

- **セッション継続**: `context_mode: "group"` でスケジュールタスクを実行すると、グループのセッションを再開する。前回の会話が見える
- **自動コンパクション**: コンテキストが 165,000 トークンを超えると自動で要約・圧縮。重要な情報は保持される
- **ファイルベースのメモリ**: kanban.md, USER.md, daily/ に永続化された情報は、コンパクションの影響を受けない。毎回ファイルを読めば状態が復元できる

**Blueclaw にとってのポイント**: 重要な状態は必ずファイルに書き出させること。会話内の「覚えておいて」だけに頼ると、コンパクション時に失われる可能性がある。CLAUDE.md に「重要な判断・発見は必ず USER.md または daily/ に記録すること」と明記する。

### 7.2 API コスト

- 各チェックインで Claude API が呼ばれる。1日あたりの呼び出し回数を見積もる:
  - 朝のセッション: 1回（+ ユーザーとの複数ラウンドのやりとり）
  - チェックイン: 宣言タスク数 × 2（開始・終了）= 最大6回
  - フォローアップ: 0-3回程度
  - 夜のセッション: 1回
  - 合計: 1日あたり **約8-12回のエージェント起動**
- `script` オプションを活用して不要な起動を減らすことでコストを最適化できる
- 1回のエージェント起動でどれだけトークンを使うかは CLAUDE.md の長さと会話の深さに依存

### 7.3 セルフチャット vs 専用グループ

Blueclaw はユーザーとの1対1の対話。NanoClaw のメインチャネル（WhatsApp セルフチャット）を使うのが最もシンプル:
- メインチャネルはトリガーワード不要（全メッセージを処理）
- 管理者権限がある（タスクの全体管理が可能）
- セルフチャットなので他人に見られない

代替: メインチャネルは管理用に残し、Blueclaw 専用のグループ（`requiresTrigger: false` で全メッセージ処理）を作る方法もある。これは将来的に「Blueclaw 以外のグループも使いたい」場合に便利。

### 7.4 チャネル選択

DESIGN.md はチャットアプリを特定していないが、NanoClaw は以下をサポート:
- **WhatsApp** (`/add-whatsapp`) — セルフチャット対応。最も一般的
- **Telegram** (`/add-telegram`) — ボット作成が簡単。通知のカスタマイズが豊富
- **Slack** (`/add-slack`) — DM をメインチャネルにできる
- **Discord** (`/add-discord`) — サーバー内でプライベートチャネルを作れる

**推奨**: 普段使いのチャットアプリを選ぶ。Blueclaw の価値は「常にそこにいる」ことなので、通知に気づきやすいアプリが最適。

## 8. CLAUDE.md の書き方のガイドライン

DESIGN.md の内容を CLAUDE.md に変換する際の注意点:

### 8.1 長さの目安

- CLAUDE.md はエージェント起動時に毎回コンテキストに読み込まれる
- 長すぎるとトークンを消費し、実際の会話に使える枠が減る
- **目標: 2,000-4,000 トークン程度**（日本語で約 3,000-6,000 文字）
- DESIGN.md の全文（約 11,000 トークン）をそのまま入れるのは多すぎる

### 8.2 構造化のコツ

- 行動原則は箇条書きで簡潔に
- 判断テーブル（回避パターン一覧等）はそのまま残す（Claude はテーブルをよく理解する）
- 具体的な発話例はいくつか残す（トーンの基準になる）
- 「なぜそうするか」の理論的背景は省略可能（Barkley モデルの説明等）

### 8.3 分割戦略

CLAUDE.md が長くなる場合:

```
groups/main/
├── CLAUDE.md          # 核心の行動原則・パーソナリティ（2,000-3,000トークン）
├── manual/
│   ├── morning.md     # 朝のセッション詳細手順
│   ├── evening.md     # 夜のセッション詳細手順
│   ├── avoidance.md   # 回避パターン検知・対応マニュアル
│   └── motivation.md  # モチベーション管理テクニック
├── kanban.md
├── USER.md
├── daily/
└── weekly/
```

CLAUDE.md に「詳細は manual/ 配下のファイルを参照すること。朝のセッション開始時は manual/morning.md を読んでから進行すること」と書いておけば、必要な時だけ詳細を読み込む形にできる。

## 9. まとめ: やることリスト

| # | やること | 所要時間の目安 |
|---|---------|-------------|
| 1 | NanoClaw をフォーク & `/setup` でセットアップ | 15-30分 |
| 2 | `.env` で `ASSISTANT_NAME=Blueclaw` に変更 | 1分 |
| 3 | `groups/main/CLAUDE.md` を Blueclaw 仕様に書き換え | 1-2時間（最重要） |
| 4 | 初回対話で kanban.md, USER.md を初期化 | 10分 |
| 5 | 朝・夜のスケジュールタスクを登録 | 対話で指示するだけ |
| 6 | 数日間使って CLAUDE.md を微調整 | 継続的 |
| 7 | 安定したらクラウドデプロイ（任意） | 1-2時間 |

**最も重要なのはステップ 3。** CLAUDE.md の品質が Blueclaw の品質に直結する。DESIGN.md の内容を適切に凝縮し、Claude が毎回の対話で正しく振る舞えるだけの情報を過不足なく記述することが、このプロジェクトの核心。

## 10. 制限事項と対策

| 制限 | 影響 | 対策 |
|------|------|------|
| macOS スリープ中はタスク不実行 | チェックインが遅延する | VPS にデプロイ or Mac の電源設定でスリープを防ぐ |
| コンテキストの自動コンパクション | 長い会話で初期の文脈が失われる | 重要情報はファイルに永続化する |
| API コスト | 頻繁なチェックインでコストが増加 | script オプションで軽量チェックを入れる |
| WhatsApp の接続切れ | メッセージを受信できなくなる | `/debug` で診断。自動再接続は組み込み済み |
| CLAUDE.md の長さ制限 | 長すぎると毎回のトークン消費が大きい | manual/ に分割して必要時に参照 |
